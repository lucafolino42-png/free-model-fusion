🚀 Quickstart

Three steps and you're running. Just install, configure, and go.

1
Install dependencies — npm install
2
Configure — cp .env.example .env and add at least one API key (e.g., GROQ_API_KEY=gsk_...)
3
Run — npm run dev (hot reload) or npm run build && npm start (production)

Then open http://localhost:3000 to see the Web UI Dashboard. Add your first API key in Secrets, then type in Chat. Or launch the Setup Wizard from the sidebar for guided onboarding.

💡 Need a key fast? Groq offers free API keys at console.groq.com/keys. Gemini is free at aistudio.google.com.

🧠 Concepts

The building blocks you need to understand before diving deeper.

🏢
Provider
An AI service hosting models (Groq, OpenRouter, Gemini, etc.). Needs a valid API key to be usable. Providers expose OpenAI-compatible HTTP endpoints.
🧩
Model
A specific AI under a provider (e.g. llama-3.3-70b-versatile). Each model has roles: expert, judge, synthesis.
⚡
Profile
Determines model selection: speed (fastest), balanced (default), quality (best), custom (your pick).
💾
Session
A conversation thread with sessionId. Stores message history, profile, web-search mode, reasoning effort, and your custom model set.
🧩
Skill
A reusable prompt fragment that adapts the fusion pipeline for specific tasks like code review, debugging, or education. Load with /skills load <name>.
🎯
Reasoning Effort
Controls how deeply models think before responding: low, medium, high, xhigh. Works with OpenRouter reasoning models.

⚙️ Fusion Pipeline — In Depth

Here's exactly what happens every time you send a message. This 12-step pipeline is the heart of Free Model Fusion.

📐 Architecture overview: Your message enters → session loads history → web search runs → query complexity analyzed → experts selected → parallel AI calls → judge evaluates → synthesis produces answer → continuation handles truncation → cost estimated → meta appended → saved to memory → returned to you.
1
Load history — The handler reads before writing, so the current turn isn't duplicated into the context. Session settings (profile, web mode, reasoning effort) are loaded.
2
Save user message — Your message is stored in the database for the current session.
3
Build context-aware prompt — If there's prior history, the last 3 user turns are summarized and prefixed to the current message. This ensures even cheap models maintain conversational context.
4
Run web search — If web mode is on or auto (and the query looks time-sensitive), Tavily is queried. Results are folded into the synthesis context.
5
Analyze query complexity — Lightweight classifier detects simple vs complex queries. Auto-selects speed for simple and quality for complex — only when session is on balanced with no override.
6
Select expert panel — Based on profile and preferred models. Provider deduplication caps each provider at maxExpertsPerProvider (default 2) for diversity.
7
Call experts in parallel — Each expert gets a diverse perspective (Technical, Practical, Analytical, Educational — MoA-inspired). Race mode may cut in early. 30-second timeout per call.
8
Run the judge — Evaluates expert responses (if a judge-eligible model is available + at least 2 expert answers). May output confidence scores.
9
Run synthesis — Produces final answer from expert answers + judge evaluation + (optional) web context. Falls back to first successful expert response if synthesis returns empty.
10
Handle truncation — If synthesis returns finish_reason: "length", the model is asked to continue. Capped by FUSION_MAX_CONTINUATIONS (default 1).
11
Estimate cost & format — Approximate USD cost is calculated based on model tiers. The response is formatted with clean visual hierarchy for Telegram (emojis, separators, compact meta footer).
12
Persist & return — Assistant message saved to the database. Returns { answer, telegramHtml, meta } with routing info, models used, cost estimate, and reasoning effort.

🛡️ Fallback Chain

❌
All experts fail
formatAllExpertsFailed enumerates which models failed and why. Check meta.errors for details.
🔑
No credentials
formatNoExpertsConfigured directs you to /addkey with example.
📭
Synthesis empty/error
Falls back to the first successful expert response. You always get an answer when at least one expert succeeded.

🏃 Race Mode

Race mode speeds up responses by not waiting for all expert models to finish. Once a minimum number of successful responses are received, the pipeline immediately proceeds to synthesis — the remaining calls continue in the background but are discarded.

ProfileRace behaviorMin responsesSpeed gain
speed / balancedRace enabled2 successful expert responses~30–50% faster
quality / customRace disabledAll expertsMaximum quality
📊 Tracking: When race mode triggers, meta.racedAhead reports how many expert calls were discarded. The Telegram meta footer also shows ⚡ Raced ahead (speed mode) when applicable.

📊 Query Complexity Analysis

The engine includes a lightweight keyword-based classifier that analyzes each message and auto-selects a routing profile for that request without changing the session's stored profile.

ClassificationProfile usedTriggers
Simplespeed"what is", "who is", "define", "how many", "capital of", short questions with ?
Complexquality"explain", "how does", "compare", "analyze", code blocks (```), multi-sentence, "refactor", "architecture"
BalancedSession defaultNo strong signal — uses the session's stored profile
When does auto-classification apply? Only when: (a) the session profile is balanced (the default), and (b) the user did not provide a per-request profile override. This keeps the system predictable while optimizing for common patterns.

The reasoning effort system controls how deeply models think before responding. Useful for multi-step reasoning, analysis, or chain-of-thought tasks.

LevelDescriptionBest for
lowFast responses, minimal reasoningSimple Q&A, quick lookups
mediumBalanced reasoning defaultGeneral conversation
highDeeper reasoning for complex tasksAnalysis, debugging, planning
xhighMaximum reasoning depth (may be slow)Research, hard proofs, architecture
# Set reasoning level (persistent per session)

⚙️ How it works: The reasoning_effort parameter is sent to models that support it. Currently routed through OpenRouter (which passes it to upstream reasoning models like OpenAI o-series). The xhigh level also enables a thinking mode with extended token budgets. The effort level is included in response meta as meta.reasoningEffort.

💰 Cost Estimation

Every response includes a rough cost estimate in meta.estimatedCostUsd. These are order-of-magnitude estimates based on model tier, not exact billing. Useful for keeping track of usage costs across providers.

🚀

⚖️
$0.0005 – $0.001
medium / strong
🎯
$0.002 – $0.005
slow / frontier
🧠
$0.005 – $0.01
very_slow / reasoning

The estimate uses the higher of the speed-based and quality-based cost for each model. Continuation calls add their own estimate. For race mode, the total includes all selected experts (even discarded ones), so it's a slight overestimate.

📱 Telegram: The cost estimate is shown in the meta footer at the bottom of each response: 💰 Estimated: $0.0012. Very small costs show as < $0.0001.

🌐 Web Search

When web search is enabled, the engine queries Tavily before answering. Results are folded into the synthesis context for fresh, accurate answers.

Setup

Set TAVILY_API_KEY in Secrets → Advanced or use /addsearchkey tavily <key>. Get a free key at tavily.com.

Modes

🔴
Off
Never search the web. Fastest responses, uses only model knowledge.
🟡
Auto
Search when the query looks time-sensitive: news, current events, dates, sports scores, weather, prices.
🟢
On
Always search before answering. Best for research-heavy use cases.
🔍 Auto-detection logic: The auto mode checks for time-sensitive keywords ("latest", "today", "news", "weather", "price"), question marks, year references (2025-2029), and app-usage questions are excluded ("how do I clear my session"). Configured in src/fusion/webSearch.ts.

Profile Detection from Model Name

The model field is parsed intelligently to determine the routing profile using a keyword scoring system:

PatternProfileExamples
Exact matchspeed / quality / etc.speed, quality, fast, deep
Prefix / SuffixUses prefix/suffixspeed-gpt, fusion-quality, my-model-speed
Keyword scoringBest match (highest score)turbo → speed, sonnet → quality, groq → speed

Tool / Function Calling

The endpoint detects tool call requests and parses tool call responses automatically. When a model responds with {"tool": "name", "arguments": {...}}, it's converted to OpenAI's tool_calls format with finish_reason: "tool_calls".

Streaming

Set stream: true for SSE (server-sent events) in standard OpenAI format. Note: the full response is generated first, then sent as chunks — true token-by-token streaming is on the roadmap.

🔧 Custom OpenAI-Compatible Providers

Any service that speaks the OpenAI Chat Completions API works — vLLM, Ollama (with OpenAI shim), LM Studio, your own self-hosted inference server, or any commercial API.

1
Add provider — Dashboard → Providers → Add Provider. Set the ID, label, and endpoint URL (must be HTTPS).
2
Add key — Go to Secrets → Add Key and paste the API key for this provider.
3
Add models — Under Models → Add Model with your endpoint's exact model field. Or use 📡 Discover to auto-fetch all models from the provider's /v1/models endpoint.
⚠️ Security: The endpoint is validated by validateProviderUrl — rejects http://, private IPs, localhost, reserved hostnames (.internal, .local), and raw IP addresses. This prevents SSRF attacks. For local development, set FUSION_ALLOW_PRIVATE_ENDPOINTS=true.

📦 Custom Models

Add any model string your provider recognizes. The fusion engine calls it with the standard OpenAI request shape and parses the response via extractContent() — it checks choices[0].message.content (OpenAI), then candidates[0].content.parts[].text (Gemini), then plain output_text / text / message.

POST <your-endpoint>
{
  "model": "<model-id-you-specified>",
  "messages": [{"role":"system","content":"..."},{"role":"user","content":"..."}],
  "max_tokens": 2500,
  "temperature": 0.3
}
💡 Role assignment: Each model has a useAs field — expert (participates in parallel panel), judge (evaluates responses), synthesis (produces final answer). A model can have multiple roles.

🔒 Security

Encryption at Rest
API keys encrypted with AES-256-GCM using FUSION_SECRET_KEY. In dev, a machine-specific key is auto-generated.

SSRF Protection

Security Headers
CSP, X-Content-Type-Options: nosniff, X-Frame-Options: DENY, Referrer-Policy on every response.

🩺 Troubleshooting

Models keep changing their API names

Run npm run check:models. It queries each provider's live model list and compares against src/providers/presets.ts. The CI workflow runs the same check weekly + on PRs that touch presets.

Or use the Discover button: In the Models view, click 📡 Discover on any provider to auto-fetch all available models from their API. New models are saved as custom entries and immediately usable.

"None of the available AI models responded successfully"

Check the per-model errors in meta.errors — most often a 401 (bad key) or 429 (rate-limited).

  • Use /listkeys to verify your keys
  • Use /providers to check which providers are enabled
  • Check that the provider has models enabled with /models
  • Verify the key hasn't expired or hit its quota in the provider's dashboard
Telegram bot replies but doesn't get my messages
  1. Confirm the token: curl -s "https://api.telegram.org/bot<TOKEN>/getMe" should return {"ok":true,...}
  2. If a webhook URL is set, the bot uses webhook mode — it won't poll. Check /api/env → TELEGRAM_WEBHOOK_URL
  3. Look at the server logs for "Telegram polling got 401/403" — the token was rotated; update it in Secrets → Advanced
Web search keeps saying "API key not configured"

Use /addsearchkey tavily <key> or set TAVILY_API_KEY in Secrets → Advanced. Get a free key at tavily.com.

Known issue (fixed): Previously, the /settings endpoint only checked config.tavilyApiKey (in-memory). If you saved the Tavily key via the UI (stored in DB), it wasn't reflected after a server restart. The endpoint now checks both in-memory config and the database credential store.

Token-budget changes in Settings don't take effect

They do, immediately. Values are saved to the DB and applied to config.expertMaxTokens on the next chat request. Verified by tests/settingsLive.test.ts. No restart needed.

If you're seeing old values, try refreshing the page. The UI reads from GET /settings which returns the DB-stored values.

My newly discovered models don't appear in the custom model selection

Known issue (fixed): Previously, clicking 📡 Discover only refreshed the Models accordion but not the Settings → Custom Model Combinations list. Now renderCustomModelsList() is called after every discovery, so the custom model picker refreshes immediately without a page reload.

"reasoning_effort is not supported" error

The reasoning_effort parameter is only sent to providers that support it (currently OpenRouter). Other providers (Groq, Cerebras, Together) reject unknown parameters with a 400 error. The error is caught gracefully. Switch to OpenRouter for reasoning effort support, or use default /reasoning medium.

The accordion toggle in Models view doesn't expand/collapse

Known issue (fixed): The modelsAccordion container was a <tbody> element. Browsers silently move <div> elements inside <tbody> outside the table structure, breaking btn.nextElementSibling in toggleProvGroup(). The container is now a <div> — toggling works correctly.

The API Keys page shows a blank page

Known issue (fixed): The view-keys HTML section was deleted in a previous session. The sidebar now redirects to Secrets (view-env) which contains the full keys table, env var editor, and add/delete key functionality. All secrets management is in one unified view.

NVIDIA NIM models don't show after adding a key

NVIDIA NIM has 0 preset models hardcoded in the codebase (unlike other providers that have presets like groq_llama3_8b). After adding your NVIDIA API key, go to the Models view, expand the NVIDIA NIM group, and click 📡 Discover to auto-fetch all available models from NVIDIA's API. They will be saved and immediately usable.
