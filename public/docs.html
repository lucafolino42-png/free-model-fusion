Quickstart

Three steps and you're running.

1. Install

npm install

2. Configure

cp .env.example .env
# Generate a secret key (required in production):
echo "FUSION_SECRET_KEY=$(openssl rand -hex 32)" >> .env
# Add at least one provider API key:
echo "GROQ_API_KEY=gsk_your_key_here" >> .env

3. Run

npm run dev   # development with hot reload
# or
npm run build && npm start   # production

Open http://localhost:3000 to see the Web UI Dashboard. Go to Secrets and add your first API key, then type in Chat. Or use the Setup Wizard from the sidebar for guided onboarding.

Concepts

Provider

An AI service that hosts models (Groq, OpenRouter, Gemini, etc.). Providers expose an OpenAI-compatible HTTP API. A provider needs at least one valid API key to be usable.

Model

A specific AI configuration under a provider (e.g. llama-3.3-70b-versatile on Groq). Each model has roles: expert judge synthesis.

Profile

How to pick the expert panel: speed (fastest), balanced (default), quality (best), custom (your pick).

Session

A conversation thread identified by sessionId. Stores message history, profile, web-search mode, reasoning effort, and your custom model set.

Skill

A reusable prompt fragment that guides model behavior. Skills adapt the fusion pipeline for specific tasks like code review, debugging, or education. Load with /skills load <name>.

Reasoning Effort

Controls how deeply models think before responding. Levels: low, medium, high, xhigh. Sent to OpenRouter and upstream reasoning models.

Endpoints

POST /chat/completions

Content-Type: application/json

{
  "model": "fusion-balanced",
  "messages": [
    {"role": "system", "content": "You are a helpful assistant."},

    {
      "object": "embedding",

Custom OpenAI-compatible providers

Any service that speaks the OpenAI Chat Completions API works. Examples: vLLM, Ollama (with OpenAI-compatible shim), LM Studio, OpenRouter, your own self-hosted inference server.

  1. Open the dashboard → Providers → Add Provider.
  2. Set the Provider ID (used internally — no spaces), a friendly Label, and the full chat-completions Endpoint URL (must be HTTPS; localhost and private IPs are rejected by default).
  3. Click Add.
  4. Go to Secrets → Add Key and paste the API key for this provider.
  5. Toggle the provider on in the Providers table.
  6. Add at least one model under Models → Add Model with your endpoint's model field exactly as it expects (e.g. llama-3.1-8b-instant, my-custom-model).

The endpoint is validated by validateProviderUrl — rejects http, private IPs, localhost, and reserved hostnames (.internal, .local). If you need to point at a local server during development, set FUSION_ALLOW_PRIVATE_ENDPOINTS=true in your env (development only).

Custom models

Add any model string your provider recognizes. The fusion engine will call it with the standard OpenAI request shape:

POST <your-endpoint>
{
  "model": "<model-id-you-specified>",
  "messages": [{"role":"system","content":"..."},{"role":"user","content":"..."}],
  "max_tokens": 2500,
  "temperature": 0.3
}

Response is parsed via extractContent: it looks at choices[0].message.content (OpenAI), then candidates[0].content.parts[].text (Gemini), then plain output_text / text / message. If your endpoint returns a custom shape, you may need to adjust extractContent in src/providers/modelClient.ts.

Reasoning Effort

The reasoning effort system controls how deeply models think before responding. This is useful for tasks that require multi-step reasoning, analysis, or chain-of-thought.

Levels

LevelDescriptionUse case
lowFast responses, minimal reasoningSimple Q&A, quick lookups
mediumBalanced reasoning (default)General conversation
highDeeper reasoning for complex tasksAnalysis, debugging, planning
xhighMaximum reasoning depth (may be slow)Research, hard proofs, architecture

Usage

# Set reasoning level (persistent per session)
/reasoning high

# View current level
/reasoning

Reasoning effort is persistent per session. Set it with /reasoning <level> and it applies to all subsequent messages. There is no per-request override — change the session level when you need different reasoning depth.

How it works

The reasoning_effort parameter is sent to models that support it. Currently, this is routed through OpenRouter, which passes it through to upstream reasoning models (OpenAI o-series, Anthropic Claude). The xhigh level also enables a thinking mode with extended token budgets for the deepest reasoning.

The reasoning effort level is included in the response meta as meta.reasoningEffort.

Fusion pipeline in depth

For every chat message, in order:

  1. Load history for the session. The handler reads before writes so the current turn isn't duplicated into the context.
  2. Build a context-aware prompt: if there's prior history, the last 3 user turns are summarized and prefixed to the current message. This ensures cheap models maintain conversational context without relying on long token histories.
  3. Run web search if web is on / auto and the query looks like it needs fresh info (heuristic: explicit time/recent keywords). Results are folded into the synthesis context.
  4. Analyze query complexity — lightweight classifier that detects simple vs complex queries. If the session is on the default balanced profile and no per-request override was provided, auto-selects speed for simple questions and quality for complex ones.
  5. Pick the expert panel based on profile and the session's preferredExperts (when profile is custom). Provider deduplication caps each provider at maxExpertsPerProvider (default 2).
  6. Call the experts in parallel with diverse perspectives (MoA-inspired). Each expert gets a slightly different role (technical, practical, analytical, educational) for more diverse responses. Per-call timeout (default 30s). Empty / timed-out / 4xx / 5xx all produce a recorded error.
  7. Race mode (speed/balanced only): proceed to synthesis once 2 experts respond successfully. Remaining in-flight calls continue in the background but are discarded. This speeds up responses by ~30–50% on average.
  8. Run the judge on the successful expert responses (if any judge-eligible model + ≥2 expert answers). The judge evaluates each response and may output confidence scores.
  9. Run synthesis with expert answers + judge evaluation + (optional) web context. If synthesis returns empty content, falls back to the first successful expert response.
  10. Continuation: if synthesis returns finish_reason: "length", ask the synthesis model to continue, then append. Capped by FUSION_MAX_CONTINUATIONS (default 1).
  11. Persist: store the assistant's final answer in the session's memory.
  12. Return: { answer, telegramHtml, meta } with cost estimates and reasoning effort level.

Fallback chain

  • All experts fail → formatAllExpertsFailed enumerates which models failed and why.
  • No credentials configured → formatNoExpertsConfigured directs the user to /addkey.
  • Synthesis empty/error → falls back to the first successful expert response.

Race Mode

Race mode speeds up responses by not waiting for all expert models to finish. Instead, once a minimum number of successful responses are received, the pipeline immediately proceeds to synthesis.

ProfileRace behaviorDefault min responses
speed / balancedRace enabled — proceed after 2 successful expert responses2
quality / customRace disabled — wait for all expertsAll

When race mode is triggered, meta.racedAhead reports how many expert calls were discarded. These calls continue in the background but their results are not used for synthesis.

Query Complexity Analysis

The engine includes a lightweight classifier that analyzes each message and auto-selects a routing profile for that request — without changing the session's stored profile.

Classification criteria

ClassificationSelected profileTriggers
SimplespeedFactual questions: "what is", "who is", "define", "how many", "capital of", short questions with ?
Complexquality"explain", "how does", "compare", "analyze", "implement", code blocks (```), multi-sentence queries, keywords like "refactor", "debug", "architecture"
BalancedSession defaultNo strong signal — uses the session's stored profile

Auto-classification only applies when: (a) the session profile is balanced (the default), and (b) the user did not provide a per-request profile override. This keeps the system predictable while optimizing for common patterns.

Cost Estimation

Every response includes a rough cost estimate in meta.estimatedCostUsd. These are order-of-magnitude estimates based on the model's speed and quality class, not exact provider billing.

Per-call cost by model tier

Speed classEst. costQuality classEst. cost
very_fast$0.00005basic$0.00005
fast$0.0001good$0.0002
medium$0.0005strong$0.001
slow$0.002frontier$0.005
very_slow$0.005reasoning$0.01

The estimate uses the higher of the speed-based and quality-based cost for each model. Continuation calls add their own estimate. For race mode, the total includes all selected experts (even discarded ones), so it's a slight overestimate.

Web search

When web is on or auto, the engine queries Tavily before answering. Results are folded into the synthesis context. Set TAVILY_API_KEY in Secrets → Advanced or use /addsearchkey tavily <key> to enable.

In auto mode, the engine heuristically decides whether to search based on query content (time-sensitive keywords, current events, recency indicators).

Telegram bot

Setup

  1. Talk to @BotFather on Telegram, send /newbot, get a token.
  2. Add TELEGRAM_BOT_TOKEN=<your-token> in Secrets → Advanced. Save. (No restart needed — the polling loop reads it live on every iteration.)
  3. Message your bot in Telegram. The first message gets the welcome (/start). Any subsequent text is treated as a chat message (session id telegram:<chat-id>, rate-limited at 1 msg / 2s per chat).

Webhooks (production)

# .env
TELEGRAM_BOT_TOKEN=...
TELEGRAM_WEBHOOK_URL=https://your-host.example.com
# Fusion auto-registers the webhook at https://your-host/telegram/webhook.

Commands in Telegram

All chat commands work in Telegram. The Telegram bot also responds to /start and /help.

Security

  • API keys are encrypted at rest using AES-256-GCM with a key derived from FUSION_SECRET_KEY.
  • SSRF protection: provider endpoints are validated to reject private IPs, localhost, and reserved hostnames.
  • Security headers on every response: CSP, X-Content-Type-Options, X-Frame-Options, Referrer-Policy (see src/server.ts).
  • Rate limiting: 100 req/min global, 20 req/min on /chat.
  • Input sanitization: error messages never include API keys or secrets.
  • Auth/authz is intentionally absent — this is a self-hosted tool meant for trusted networks. Put it behind a reverse proxy (Caddy / nginx / Cloudflare Tunnel) if exposing it.

Troubleshooting

Models keep changing their API names

Run npm run check:models. It queries each provider's live model list and compares against src/providers/presets.ts. The CI workflow .github/workflows/model-freshness.yml runs the same check weekly + on PRs that touch presets.

"None of the available AI models responded successfully"

Check the per-model errors in meta.errors — most often a 401 (bad key) or 429 (rate-limited). Use /listkeys to verify your keys, and /providers to check which providers are enabled.

Telegram bot replies but doesn't get my messages

  1. Confirm the token: curl -s "https://api.telegram.org/bot<TOKEN>/getMe" should return {"ok":true,...}.
  2. If a webhook URL is set, the bot is using webhook mode — it won't poll. Check /api/env → TELEGRAM_WEBHOOK_URL.
  3. Look at the server logs for "Telegram polling got 401/403" — the token was rotated; update it in Secrets → Advanced.

Token-budget changes in Settings don't take effect

They do, immediately. The values are saved to the DB and applied to config.expertMaxTokens on the next chat request (verified by tests/settingsLive.test.ts).

Web search keeps saying "API key not configured"

Use /addsearchkey tavily <key> or set TAVILY_API_KEY in Secrets → Advanced. Get a free key at tavily.com.

"reasoning_effort is not supported" error

The reasoning_effort parameter is only sent to providers that support it (currently OpenRouter). If you're using a different provider that rejects unknown parameters, the error is caught and handled. Switch to OpenRouter for reasoning effort support, or set /reasoning medium to use the default.
